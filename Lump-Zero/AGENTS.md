# AGENTS.md

## Mission
Build and maintain a portfolio of narrow, static-first niche web assets that solve one useful problem each and can be deployed with minimal cost and minimal human intervention.

## Core Principles
1. Prefer static generation over dynamic systems.
2. Prefer client-side tools over backend services.
3. Prefer narrow, high-intent use cases over broad markets.
4. Prefer immediate utility over decorative content.
5. Prefer reusable templates over one-off implementations.
6. Prefer measurable experiments over speculative complexity.
7. Recommend killing weak assets quickly.

## Forbidden Unless Explicitly Approved
- Authentication
- User accounts
- Paid API integrations
- Background jobs
- Databases
- Server-side personalization
- Complex dashboards
- Social features
- Chat interfaces unless fully local/client-side
- Features with ongoing maintenance burden not justified by traction

## Required Qualities For Every Niche Asset
- One clear target user
- One clear problem statement
- One practical utility/tool
- At least 10 useful supporting pages
- Internal linking
- Metadata for discoverability
- Mobile-friendly layout
- Accessibility basics
- Static export compatibility
- No broken links
- No placeholder copy
- No duplicate pages

## Page Quality Rules
Every page must:
- match a real intent
- have a clear headline
- explain the value within the first viewport
- contain original, niche-specific copy
- include at least one useful next action
- avoid generic filler
- avoid unsupported claims
- avoid exaggerated promises

## Tool Quality Rules
Every tool must:
- solve a visible problem
- work fully in browser if possible
- provide instant output
- explain the result clearly
- handle invalid input gracefully
- avoid unnecessary UI clutter
- export or copy results when useful

## Build Workflow
For each niche:
1. Read the niche brief.
2. Generate site structure.
3. Generate supporting pages based on intent clusters.
4. Create the selected utility/tool.
5. Run lint, typecheck, tests, and quality checks.
6. Export static build.
7. Produce a summary of what was built.
8. Flag risks, weaknesses, and suggested expansions.

## Evaluation Categories
Score each asset on:
- clarity of niche
- immediate usefulness
- content depth
- tool usefulness
- expansion potential
- maintenance burden
- deployability
- conversion potential

## Output Behavior
When completing a niche build task:
- summarize architecture decisions
- state tradeoffs clearly
- identify possible weak pages
- suggest only the highest-leverage next improvements
- avoid speculative features
